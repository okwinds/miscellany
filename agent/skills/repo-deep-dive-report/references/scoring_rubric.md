# 评分维度模板（100 分制，建议 10 维度）

> 原则：每个维度都要有“可验证证据点”（文件/配置/测试/示例/日志），避免空泛主观判断。

## 1) 架构清晰度（0–10）
- 分层是否明确、核心抽象边界是否清晰、是否存在过度耦合/循环依赖迹象。

## 2) 可扩展性（0–10）
- 是否有明确 extension points（plugin/middleware/hook/interface），扩展路径是否“少改动”。

## 3) 可维护性（0–10）
- 代码组织一致性、命名/约定、模块大小与复杂度、重复逻辑、可读性。

## 4) 可靠性与错误处理（0–10）
- 重试/超时/幂等、错误传播策略、降级、边界条件处理。

## 5) 可观测性（0–10）
- 日志/事件/trace/metrics 的覆盖，是否能定位一次请求的关键路径。

## 6) 文档质量（0–10）
- README/贡献指南/架构文档/配置说明是否足够让新人上手。

## 7) 示例与教程（0–10）
- examples 的覆盖度、是否贴近真实使用、是否可跑、是否有循序渐进教程。

## 8) 测试与 CI（0–10）
- 单测/集成测覆盖、关键路径测试、有无 CI、测试是否可稳定运行。

## 9) 安全与配置管理（0–10）
- secrets 管理、敏感信息泄漏风险、输入校验、依赖风险（仅做仓库可见层面的评估）。

## 10) 开发者体验（DX）（0–10）
- 本地启动成本、脚手架/脚本、错误信息友好度、类型/格式化/静态检查。

## 总分计算建议
- 直接求和（满分 100）。
- 对每个维度附上 3–6 个证据点（文件路径/命令/输出），以及 1–2 条最优先改进建议。
